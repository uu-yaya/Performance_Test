# 桌宠功能点总结

## 1. 个人调研总结

当前陪伴类桌宠的趋势，不是简单地“加一个聊天机器人”，而是把 Agent 能力逐步融合进桌宠里。

早期桌宠的行为大多是写死的，核心体验是可爱形象、固定动作、简单点击反馈，以及基础 QA。它可以陪伴，但更像一个有互动反馈的桌面摆件。进化后的桌宠，本质上会越来越像一个“可视化 Agent”：它不只是回答问题，而是能理解上下文、记住用户、识别意图、调用工具，并通过形象、动作、表情和话术把自己的状态表达出来。

这和黑盒 Agent 很不一样。黑盒 Agent 经常让人困惑，因为用户不知道它在想什么、为什么这样做、下一步要干嘛。但桌宠天然有一个可视化身体，它可以把 Agent 的关键状态展示出来，比如“我注意到了什么”“我准备做什么”“我为什么这么说”“我学到了什么”。这让 Agent 的推理和行动不再只是后台逻辑，而变成一种可被感知的陪伴体验。

Agent 最关键的能力可以概括为：LLM 推理 + 记忆 + 反思 + 工具调用。放到桌宠场景里，这套能力会让桌宠从“会聊天的角色”进化成“能理解用户并采取行动的伙伴”。它能更准确地理解用户意图，执行更丰富的动作，也能在长期互动中形成自己的陪伴风格。

我觉得最关键的有两点：

- 如何把 Agent 所想可视化。
    单纯配置桌宠行为其实很常见，但这种配置感很强，容易显得“没灵魂”。真正有生命感的地方在于，它能不能基于记忆和互动经验，自己学会新的表达、新的动作、新的话术，甚至逐渐形成一种“我和这个用户一起长出来的性格”。
    比如 Codex 可以根据一个 “Hatch Patch” 技能和一张图片搭建出新的桌宠，包括形象和 action，这说明桌宠的行为资产本身是可以被动态生成和持续更新的。进一步看，Agent 完全可以基于沉淀下来的记忆，在低打扰的后台周期里生成新的行为候选，比如在用户睡觉的时候偷偷进化，悄悄学会新的撒娇方式、生气跺脚、鼓励话术，或者某种只属于这个用户的互动梗。
    当然这种“进化”最好不是完全偷偷发生，而是低打扰、可预览、可撤回的。用户应该能感受到宠物在成长，但也要知道它学了什么、为什么学、哪些记忆被使用了。这样桌宠才会既有生命感，又不会让人觉得失控。

- 如何吸引用户持续使用桌宠
    我觉得单纯依靠“自进化”或者宣称“越用越懂你”已经不够有差异化了，因为市面上很多陪伴 Agent 都在用这个宣传。真正值得突破的点可能不是“它越来越懂我”，而是“它越来越懂我们”。桌宠最早吸引人的地方，本质上不是功能，而是人会天然对一个由自己参与养成、陪伴成长的生命体产生情感投入。现实里，宠物或者小孩很多时候也不是由一个人独自抚养的，它们反而经常会成为两个人关系里的情感联结。所以除了好友联排、群聊这些已有的关系玩法之外，其实还可以增加“共同抚养一只宠物”的模式。
    以情侣共同抚养一只桌宠为例，这只宠物可以同时出现在两个人的桌面上，记住两个人各自的偏好、相处方式、共同经历过的游戏高光、纪念日，甚至是他们一路走来的关系记忆。此时它就不再是一个简单的桌宠，而是两个人关系的美好象征。它也可以成为关系里的润滑剂，比如帮忙传达一些不好意思直接说出口的话，在双方情绪不好的时候做做适当的提醒、稍微幽默一下缓解氛围，或者用共同回忆帮助两个人重新唤醒美好回忆（maybe大家都很吃这套）。随着宠物不断自进化、越来越理解两个人，它不只是让用户更喜欢这个宠物，也可能让两个人因为这个宠物而更加了解彼此。

## **3. 功能层级**

| **层级**    | **功能组**                                                                       | **产品含义**                               |
| --------- | ----------------------------------------------------------------------------- | -------------------------------------- |
| 智能核心      | LLM / 意图识别 / 记忆 / 反思 / RAG / tools                                            | 决定桌宠“懂不懂用户、会不会成长、能不能做事”。               |
| 感知输入      | game event / screenshot / screen / audio / mouse / chat                       | 决定桌宠“为什么此刻回应”。                         |
| 行动输出      | text / voice / expression / motion / card / notification / tool action        | 决定桌宠“怎样表达和行动”。                         |
| 具身外壳      | avatar / animation / overlay / skin / voice / idle behavior                   | 决定桌宠“像不像一个活着的角色”。                      |
| 非 AI 留存循环 | idle presence / mouse interaction / care loop / skins / onboarding            | 决定桌宠“不调用 AI 时是否仍然好玩、好看、好留下来”。          |
| 关系型留存循环   | shared pet / shared memory / co-care / relationship recap / message mediation | 决定桌宠能否从“我的陪伴”变成“我们的关系资产”，带来双人或小队级回访动力。 |
| 控制与安全     | permissions / quiet / memory control / data path / fallback / logs            | 决定用户和开发者是否敢让它常驻。                       |
| 开发者平台     | schema / templates / preview / debugger / SDK docs                            | 决定它能否赋能多个游戏，而不是一次性 demo。               |
